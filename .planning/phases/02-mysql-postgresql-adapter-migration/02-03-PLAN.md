# 02-03: Move Row, CRUD, Explain, And Export SQL Construction Behind Adapter APIs

<objective>
Finish the MySQL/PostgreSQL adapter migration by moving remaining SQL construction for row browsing, row edits, explain, streaming export, and SQL export inserts behind adapter/dialect APIs. Remove stale protocol-based helper functions while preserving existing REST behavior and test coverage.
</objective>

<threat_model>
This plan touches SQL builders that interpolate identifiers and render exported data. The implementation must keep identifier quoting adapter-owned, keep values parameterized for row edits, preserve row-browse metadata validation before building SQL, and avoid introducing SQL injection in SQL export output. Read/write approval and audit paths must remain unchanged.
</threat_model>

<tasks>

<task id="02-03-01" type="execute">
<title>Expand dialect SQL builder surface for CRUD, explain, and export</title>
<read_first>
- `internal/dbquery/adapter.go`
- `internal/dbquery/crud.go`
- `internal/dbquery/processes.go`
- `internal/api/db_handler.go`
</read_first>
<action>
Extend `Dialect` or add an adapter-owned SQL builder interface for the remaining pure SQL builders.

Concrete target methods should cover:
- `BuildRowsSQL(schema, table, orderBy, orderDir string, limit, offset int) (string, error)` from Phase 1.
- `BuildInsertSQL(schema, table string, cols []string) (string, error)`.
- `BuildUpdateSQL(schema, table string, setCols, keyCols []string) (string, error)`.
- `BuildDeleteSQL(schema, table string, keyCols []string) (string, error)`.
- `BuildExplainSQL(statement string, analyze bool) (string, error)`.
- `BuildExportSelectSQL(schema, table string, limit int) (string, error)`.
- `BuildExportInsertSQL(schema, table string, cols []ColumnMeta, row []any) (string, error)` or an equivalent service-level helper that receives an adapter dialect.

Keep value parameterization for CRUD. Do not turn row edit values into string literals.
</action>
<acceptance_criteria>
- `internal/dbquery/adapter.go` exposes builder methods covering CRUD, explain, export SELECT, and SQL export inserts or equivalent adapter-owned helpers.
- Existing `BuildRowsSQL` behavior remains covered.
- CRUD builders still return placeholders, not interpolated values.
- SQL export insert rendering uses adapter dialect identifier quoting.
</acceptance_criteria>
<verify>
- `go test ./internal/dbquery -run 'Test.*SQL'`
</verify>
</task>

<task id="02-03-02" type="execute">
<title>Move CRUD and explain SQL construction out of service switches</title>
<read_first>
- `internal/dbquery/crud.go`
- `internal/dbquery/adapter_mysql.go`
- `internal/dbquery/adapter_postgres.go`
- `internal/api/db_handler.go`
</read_first>
<action>
Refactor row edit methods and `Explain` to call adapter/dialect builders.

Concrete target behavior:
- `UpdateRow` validates payload lengths and row keys, then asks adapter/dialect for the `UPDATE` statement.
- `InsertRow` validates payload lengths, then asks adapter/dialect for the `INSERT` statement.
- `DeleteRow` validates key payload, then asks adapter/dialect for the `DELETE` statement.
- `Explain` validates/get pool as before, checks adapter `Explain` / `ExplainAnalyze` capabilities, asks adapter/dialect for explain SQL, then runs the query path.
- Existing handler-level safety check for `/db/explain` remains in `internal/api/db_handler.go`.
- Remove stale `quoteIdent(model.NodeProtocol)` and `placeholder(model.NodeProtocol, int)` helpers if no longer used.
</action>
<acceptance_criteria>
- `internal/dbquery/crud.go` no longer switches on `pl.protocol` for explain.
- `internal/dbquery/crud.go` no longer defines or uses protocol-based `quoteIdent` and `placeholder` helpers.
- MySQL CRUD SQL uses `?` placeholders and backtick identifiers.
- PostgreSQL CRUD SQL uses `$1`, `$2`, etc. and double-quoted identifiers.
- PostgreSQL explain SQL preserves existing `EXPLAIN (ANALYZE, BUFFERS, VERBOSE)` behavior when `analyze` is true.
- MySQL explain SQL preserves existing `EXPLAIN FORMAT=TREE` and `EXPLAIN ANALYZE FORMAT=TREE` behavior.
</acceptance_criteria>
<verify>
- `go test ./internal/dbquery ./internal/api`
</verify>
</task>

<task id="02-03-03" type="execute">
<title>Move export SELECT and SQL INSERT rendering behind adapters</title>
<read_first>
- `internal/dbquery/processes.go`
- `internal/api/db_handler.go`
- `internal/dbquery/adapter_mysql.go`
- `internal/dbquery/adapter_postgres.go`
</read_first>
<action>
Refactor streaming export so table SELECT construction and SQL-format insert rows use adapter/dialect APIs.

Concrete target behavior:
- `StreamExport` or its private helper asks the adapter/dialect for export SELECT SQL instead of calling old `quoteIdent(pl.protocol)`.
- SQL export format no longer hardcodes PostgreSQL double-quoted identifiers in `internal/api/db_handler.go`.
- Either `DBHandler` obtains an adapter-owned row renderer through `dbquery.Service`, or `StreamExport` supports a mode where the service renders SQL insert lines. Prefer the smallest change that keeps CSV and JSONL behavior unchanged.
- CSV escaping, JSONL object rendering, and `sqlLiteral` value escaping remain behavior-compatible unless moved into a shared adapter-aware helper with tests.
- SQL export `INSERT INTO <schema>.<table> (<columns>) VALUES (...)` uses MySQL backticks for MySQL and double quotes for PostgreSQL.
</action>
<acceptance_criteria>
- `internal/dbquery/processes.go` no longer calls `quoteIdent(pl.protocol)`.
- `internal/api/db_handler.go` no longer hardcodes `"` identifier quoting for SQL export inserts unless it delegates through a PostgreSQL dialect selected by the service.
- MySQL export SELECT builder quotes schema/table with backticks.
- PostgreSQL export SELECT builder quotes schema/table with double quotes.
- SQL export insert tests cover identifier escaping, string literal escaping, `NULL`, bools, and at least one JSON-marshaled fallback value.
</acceptance_criteria>
<verify>
- `go test ./internal/api ./internal/dbquery`
</verify>
</task>
