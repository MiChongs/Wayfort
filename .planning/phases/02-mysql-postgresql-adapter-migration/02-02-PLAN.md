# 02-02: Move Metadata, Structure, Process, And DDL Behavior Into Adapters

<objective>
Remove service-level MySQL/PostgreSQL switches for schema browsing, table structure, process list, process cancel, and table DDL by delegating those operations to adapter-owned methods while preserving existing response shapes and SQL behavior.
</objective>

<threat_model>
This plan changes the code that drives the DB Studio schema tree and operational process controls. The main risks are UI regressions from changed DTO shape or object kind names, permission regressions from silent unsupported responses, and unsafe process kill SQL construction. Unsupported capabilities should fail clearly or be gated by adapter capabilities rather than returning misleading empty results.
</threat_model>

<action>
Extend the adapter contract with methods or grouped subinterfaces for metadata, structure, and process behavior.

Concrete target API should cover:
- `ListDatabases(ctx context.Context, db *sql.DB) ([]string, error)`.
- `LoadSchema(ctx context.Context, db *sql.DB) (*SchemaInfo, error)`.
- `LoadColumns(ctx context.Context, db *sql.DB, schema, table string) ([]ColumnInfo, error)`.
- `LoadIndexes(ctx context.Context, db *sql.DB, schema, table string) ([]IndexInfo, error)`.
- `LoadForeignKeys(ctx context.Context, db *sql.DB, schema, table string) ([]ForeignKeyInfo, error)`.
- `LoadTableStats(ctx context.Context, db *sql.DB, schema, table string) (*TableStats, error)`.
- `LoadTableDDL(ctx context.Context, db *sql.DB, schema, table string) (string, error)`.
- `ListProcesses(ctx context.Context, db *sql.DB) ([]ProcessInfo, error)`.
- `CancelProcess(ctx context.Context, db *sql.DB, pid int64) (bool, error)`.

If direct `Adapter` methods become too large, use grouped interfaces while keeping `Service` delegation simple.
</action>

- New method signatures accept `*sql.DB` or an equivalent minimal query interface, not `*pool` unless strongly justified.
- Unsupported behavior is represented by capabilities and explicit errors, not silent nil values in service wrappers.
- Existing DTO types remain in `internal/dbquery` and are not renamed.
</acceptance_criteria>

<task id="02-02-04" type="execute">
<title>Move process list and cancel behavior behind adapters</title>
<read_first>
- `internal/dbquery/processes.go`
- `internal/dbquery/adapter_mysql.go`
- `internal/dbquery/adapter_postgres.go`
- `internal/api/db_handler.go`
</read_first>
<action>
Refactor `ListProcesses` and `CancelProcess` so service methods delegate to adapter methods and check capabilities.

Concrete target behavior:
- PostgreSQL list process SQL remains based on `pg_stat_activity` and excludes `pg_backend_pid()`.
- PostgreSQL cancel remains parameterized as `SELECT pg_cancel_backend($1)`.
- MySQL list process SQL remains based on `information_schema.PROCESSLIST` and excludes `CONNECTION_ID()`.
- MySQL cancel keeps validating/formatting `pid` as an integer and emits `KILL QUERY <pid>` only from adapter code.
- If an adapter reports no `Processes` or `KillProcess` capability, service returns a clear unsupported error.
</action>
<acceptance_criteria>
- `internal/dbquery/processes.go` no longer contains process-list or process-cancel protocol switches.
- `internal/dbquery/processes.go` no longer references `model.NodeProtoMySQL` or `model.NodeProtoPostgres` except possibly in comments scheduled for removal.
- PostgreSQL cancel remains parameterized and does not interpolate untrusted SQL text.
- MySQL cancel only interpolates an `int64` pid.
</acceptance_criteria>
<verify>
- `go test ./internal/dbquery ./internal/api`
</verify>
</task>
