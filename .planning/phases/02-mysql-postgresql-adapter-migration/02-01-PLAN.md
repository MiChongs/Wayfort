# 02-01: Move Connection And Dialect Behavior Into MySQL/PostgreSQL Adapters

<objective>
Make adapter implementations own driver connection construction, database/default-schema selection, and dialect identity while keeping `dbquery.Service` responsible for access checks, credential loading, proxy-chain setup, pool lifecycle, and public method signatures.
</objective>

<threat_model>
This plan touches credential-bearing connection setup. The implementation must not move credential repository access, KMS secret opening, asset access checks, or proxy-chain construction into adapters. Adapters should receive only the already-authorized connection parameters and a chain-backed dial callback. MySQL global dialer cleanup must remain reliable on open failure, ping failure, idle eviction, and service shutdown.
</threat_model>

<task id="02-01-01" type="execute">
<title>Extend adapter contract with connection request and open result</title>
<read_first>
- `internal/dbquery/adapter.go`
- `internal/dbquery/service.go`
- `internal/dbquery/adapter_mysql.go`
- `internal/dbquery/adapter_postgres.go`
</read_first>
<action>
Extend `internal/dbquery/adapter.go` with connection-oriented types that do not depend on `webssh.Gateway`, repositories, or Gin.

Concrete target API shape:
- Add a `DialContextFunc` type or equivalent: `func(context.Context, string, string) (net.Conn, error)`.
- Add a `ConnectRequest` struct containing `NodeID`, `UserID`, `Host`, `Port`, `Username`, `Password`, `RequestedDatabase`, `ProtoOptions`, and `DialContext` or equivalently named fields.
- Add an `OpenedConnection` or `ConnectResult` struct containing `DB *sql.DB`, `Cleanup func()`, and optional debug metadata such as `Database` if useful.
- Add adapter methods equivalent to `ResolveDatabase(requested, protoOptions string) string` and `Open(ctx context.Context, req ConnectRequest) (*OpenedConnection, error)`.
- Keep `Protocol()`, `Capabilities()`, and `Dialect()` from Phase 1.
</action>
<acceptance_criteria>
- `internal/dbquery/adapter.go` defines a connection request type that includes host, port, username, password, requested database, proto options, and a dial callback.
- The connection request type does not import or reference `webssh`, Gin, repository types, or approval types.
- The adapter contract exposes a method for database/default-schema resolution.
- The adapter contract exposes a method for opening a `*sql.DB` or equivalent opened connection result.
- `Capabilities` and `Dialect` remain available to existing callers.
</acceptance_criteria>
<verify>
- `go test ./internal/dbquery -run TestDefaultRegistry`
</verify>
</task>

<task id="02-01-02" type="execute">
<title>Move PostgreSQL connection construction into postgres adapter</title>
<read_first>
- `internal/dbquery/service.go`
- `internal/dbquery/adapter_postgres.go`
- `go.mod`
</read_first>
<action>
Move PostgreSQL-specific open logic from `Service.build` into the PostgreSQL adapter.

Concrete target behavior:
- PostgreSQL adapter imports `github.com/jackc/pgx/v5` and `github.com/jackc/pgx/v5/stdlib`.
- PostgreSQL adapter resolves an empty requested database through proto options or falls back to `postgres`.
- PostgreSQL adapter builds a `pgx.ConnConfig` with host, port, username, password, database, no TLS config, and a `DialFunc` that calls the supplied chain dial callback.
- PostgreSQL adapter returns `pgxstdlib.OpenDB(*connCfg)` and a nil/no-op cleanup.
- Service no longer contains PostgreSQL driver setup code after this migration.
</action>
<acceptance_criteria>
- `internal/dbquery/adapter_postgres.go` imports `pgx` and `pgxstdlib` or equivalent PostgreSQL driver packages.
- `internal/dbquery/service.go` no longer imports `github.com/jackc/pgx/v5` or `github.com/jackc/pgx/v5/stdlib`.
- Empty PostgreSQL requested database resolves to proto options database/dbname/schema when present, otherwise `postgres`.
- PostgreSQL adapter connection code uses the supplied dial callback, not `net.Dial` directly.
</acceptance_criteria>
<verify>
- `go test ./internal/dbquery -run 'Test(Postgres|DefaultRegistry)'`
</verify>

<task id="02-01-03" type="execute">

<read_first>
- `internal/dbquery/service.go`
- `internal/dbquery/adapter_mysql.go`
- `internal/dbquery/processes.go`
</read_first>
<action>
Move MySQL-specific open logic from `Service.build` into the MySQL adapter and return cleanup that deregisters the global MySQL dialer name.

Concrete target behavior:
- MySQL adapter imports `github.com/go-sql-driver/mysql`.
- MySQL adapter resolves an empty requested database through proto options, otherwise keeps an empty database to preserve current no-default-schema behavior.
- MySQL adapter registers a unique dialer name that includes node ID, user ID, and a monotonic/unique suffix.
- MySQL adapter uses the supplied dial callback for all actual network dialing.
- MySQL adapter builds the same effective DSN options as Phase 1: `parseTime=true&loc=Local&charset=utf8mb4`.
- MySQL adapter returns cleanup that calls `DeregisterDialContext` exactly once per registered dialer.
</action>

- `internal/dbquery/service.go` no longer imports `github.com/go-sql-driver/mysql`.
- `pool` no longer needs a MySQL-specific `dialerName` field, or that field is unused and removed in the same plan.
- MySQL cleanup is called on `sql.Open` failure where applicable, ping failure, race-discarded pools, idle eviction, and service shutdown.
- MySQL DSN tests or helper tests assert the expected dialer network, address, database segment, and query options.
</acceptance_criteria>

<task id="02-01-04" type="execute">
<title>Simplify Service.build to delegate connection opening through the adapter</title>
<read_first>
- `internal/dbquery/service.go`
- `internal/dbquery/rows.go`
- `internal/dbquery/service_test.go`
</read_first>
<action>
Refactor `Service.build` so the only protocol-specific behavior is adapter lookup. Keep asset checks, node lookup, credential lookup, secret opening, proxy-chain resolution, and ping in the service.

Concrete target behavior:
- `Service.build` looks up `adapter := registry.Get(node.EffectiveProtocol())` as Phase 1 already does.
- After decrypting credentials and building the proxy chain, `Service.build` constructs a `ConnectRequest` and calls `adapter.Open`.
- `Service.build` sets connection pool limits on the returned `*sql.DB` as before.
- `Service.build` pings the DB; if ping fails, it closes `DB`, calls adapter cleanup, and releases the proxy chain.
- `pool` stores `adapter` and a generic cleanup function, not a MySQL-specific cleanup field.
- `evictIdle`, `closeAll`, and race-discard cleanup call the generic cleanup.
</action>
<acceptance_criteria>
- `internal/dbquery/service.go` has no `switch node.EffectiveProtocol()` connection branch.
- `internal/dbquery/service.go` has no direct references to MySQL or PostgreSQL driver package names.
- `Service.build` calls an adapter `Open` or equivalent method.
- Pool cleanup calls a generic adapter cleanup callback on all close paths.
- Existing access gate tests in `internal/dbquery/service_test.go` still pass.
</acceptance_criteria>
<verify>
- `go test ./internal/dbquery`
</verify>
</task>
