# 01-01: Add DB Node Access Gate And SQL Safety Baseline

<objective>
Add the missing asset-access gate for DB Studio and DB CLI before credential/session access, then harden the DB Studio read/write SQL classifier against obvious bypasses while preserving existing approval and audit behavior.
</objective>

<threat_model>
Protected assets are database node credentials, active database sessions, and data returned by DB Studio/DB CLI. Threats addressed in this plan are authenticated users guessing node IDs, continuing to use cached DB pools after grants are revoked, and sending write-capable SQL through the read-only `/db/query` or `/db/explain` paths. The implementation must fail closed if the asset resolver is not configured and must not remove existing approval or audit gates.
</threat_model>

<action>
Modify `internal/dbquery/service.go` to depend on `internal/asset` and enforce `asset.ActionConnect` inside `Service.getOrOpen` before checking `s.pools` or calling `s.build`. Use a private helper named `requireNodeAccess(ctx context.Context, userID, nodeID uint64) error` or an equivalent minimal helper.

Concrete target behavior:
- `Service` has an access checker dependency. Prefer a small package-local interface such as `type accessChecker interface { Check(context.Context, uint64, uint64, string) (bool, error) }` so tests can use fakes; `*asset.Resolver` must satisfy it.
- `New` or a minimal setter wires the resolver from `cmd/wayfort/main.go` using the existing `assetResolver` created there.
- `requireNodeAccess` returns an error if the resolver is nil.
- `requireNodeAccess` calls `s.asset.Check(ctx, userID, nodeID, asset.ActionConnect)`.
- If the check returns false, return a permission error before any credential lookup or pool reuse.
- `getOrOpen` calls the helper before reading `s.pools[key]`.
- Existing `poolKey(nodeID, userID, database)` behavior remains unchanged.
</action>

<task id="01-01-03" type="execute">
<title>Harden DB Studio read-only SQL classification</title>
<read_first>
- `internal/api/db_handler.go`
- `internal/api/db_handler_test.go` if it exists
- `internal/dbquery/crud.go`
- `internal/approval/enforcement.go`
</read_first>
<action>
Refactor `isReadOnlySQL` in `internal/api/db_handler.go` into a small classifier that preserves the existing safe heads but rejects obvious write-capable bypasses.

Concrete target behavior:
- Keep accepting simple `SELECT`, `SHOW`, `DESCRIBE`, `DESC`, and `VALUES` statements.
- Permit one trailing semicolon, but reject additional statements such as `SELECT 1; DELETE FROM t`.
- Reject top-level write heads including `INSERT`, `UPDATE`, `DELETE`, `MERGE`, `CREATE`, `ALTER`, `DROP`, `TRUNCATE`, `CALL`, `DO`, `COPY`, `GRANT`, and `REVOKE`.
- Reject `WITH` statements when the normalized SQL contains write heads such as `INSERT`, `UPDATE`, `DELETE`, or `MERGE` before the final read query.
- Reject `EXPLAIN ANALYZE` through `isReadOnlySQL`; `/db/explain` may still expose `analyze` through its explicit boolean after validating the inner statement.
- Reject SQL containing `INTO OUTFILE`, `INTO DUMPFILE`, or PostgreSQL-style `SELECT ... INTO` table creation.
- Reject known side-effect process functions in read paths, at minimum `pg_terminate_backend`, `pg_cancel_backend`, and `pg_reload_conf`.
- Keep the function private unless tests require package-level access.
</action>
<acceptance_criteria>
- `internal/api/db_handler.go` no longer has comments saying writable CTE false positives are accepted.
- `isReadOnlySQL("SELECT 1")` remains true.
- `isReadOnlySQL("SELECT 1; DELETE FROM users")` is false.
- `isReadOnlySQL("WITH x AS (DELETE FROM t RETURNING *) SELECT * FROM x")` is false.
- `isReadOnlySQL("EXPLAIN ANALYZE SELECT * FROM users")` is false.
- `isReadOnlySQL("SELECT * INTO new_table FROM users")` is false.
- `isReadOnlySQL("SELECT pg_terminate_backend(123)")` is false.
</acceptance_criteria>
<verify>
- `go test ./internal/api -run TestIsReadOnlySQL`
</verify>
</task>
