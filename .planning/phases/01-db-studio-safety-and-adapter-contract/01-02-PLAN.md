# 01-02: Define Relational Adapter Registry And Row SQL Seam

<objective>
Introduce the minimal relational adapter registry and capability contract that later phases will use for MySQL/PostgreSQL migration and Dameng support, then remove the current DB handler row-browse dialect inference by routing row SELECT construction through the DB query service/dialect seam.
</objective>

<threat_model>
This plan does not add a new external protocol, but it changes the SQL construction seam used by row browsing. The main risks are SQL injection through identifiers, accidental behavior regression for MySQL/PostgreSQL row browsing, and designing a contract too vague for Dameng. The implementation must keep metadata validation before identifier interpolation and must use dialect-owned quoting rather than string concatenation from unvalidated request parameters.
</threat_model>

<task id="01-02-02" type="execute">
<title>Add MySQL and PostgreSQL adapter implementations that preserve current dialect behavior</title>
<read_first>
- `internal/dbquery/crud.go`

<action>
Add `internal/dbquery/adapter_mysql.go` and `internal/dbquery/adapter_postgres.go`. Implement adapters that return capabilities matching current behavior and dialect methods matching current quoting/placeholders.

Concrete target behavior:
- MySQL dialect `QuoteIdent("a`b")` returns a backtick-quoted identifier with doubled backticks.
- PostgreSQL dialect `QuoteIdent("a\"b")` returns a double-quoted identifier with doubled double quotes.
- PostgreSQL `Placeholder(1)` returns `$1`; MySQL `Placeholder(1)` returns `?`.
- MySQL capabilities set `last_insert_id: true` and `database_scope` to schema-like semantics.
- PostgreSQL capabilities set `last_insert_id: false` and `database_scope` to catalog/connection-bound semantics.
- MySQL and PostgreSQL capabilities set `export: true` because current `StreamExport` supports both.
- `BuildRowsSQL` validates `limit` and `offset` are non-negative or returns an error; it quotes schema/table/order identifiers through the dialect and emits `SELECT * FROM <schema>.<table> ... LIMIT <n> OFFSET <n>`.
</action>
<acceptance_criteria>
- `internal/dbquery/adapter_mysql.go` exists.
- `internal/dbquery/adapter_postgres.go` exists.
- MySQL adapter capabilities include `LastInsertID: true` or JSON-equivalent field value.
- PostgreSQL adapter capabilities include `LastInsertID: false` or omit only if false is the zero value.
- MySQL and PostgreSQL adapter capabilities include `Export: true` or JSON-equivalent field value.
- MySQL row SQL contains backtick-quoted schema and table names.
- PostgreSQL row SQL contains double-quoted schema and table names.
</acceptance_criteria>
<verify>

<action>
Add a service method named `BuildRowsSQL` or similarly explicit API on `dbquery.Service`:

`BuildRowsSQL(ctx context.Context, nodeID, userID uint64, database, schema, table, orderBy, orderDir string, limit, offset int) (string, error)`

Concrete target behavior:
- The service method opens or reuses the pool once, then performs a metadata validation step before building SQL. Prefer extracting the current `LoadColumns` protocol switch into a private helper like `loadColumnsForPool(ctx, pl, schema, table)` so `BuildRowsSQL` can reuse the same `pl` instead of calling `LoadColumns` and then `getOrOpen` again.
- It builds `knownCols` from returned columns and rejects `orderBy` if not present.
- It uses the current pool adapter's `Dialect().BuildRowsSQL(...)` to quote schema/table/order identifiers.
- `DBHandler.Rows` calls `h.Svc.BuildRowsSQL(...)` instead of `h.buildRowsSQL(...)`.
- Delete `DBHandler.buildRowsSQL` or reduce it to a one-line delegation that no longer infers dialect from column types.
- The comment that says `Svc has no public protocol getter` is removed.
</action>

- `DBHandler.Rows` calls `h.Svc.BuildRowsSQL` or equivalent service method.
- Row SQL still validates `order_by` against table columns before quoting.
- `BuildRowsSQL` does not perform two separate pool opens/lookups for the same call unless the implementation documents why that is unavoidable.
</acceptance_criteria>
