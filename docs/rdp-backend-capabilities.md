# RDP Backend Capabilities

| Capability | Status | Notes / required before enabling |
| --- | --- | --- |
| Classic bitmap display | Enabled | `desktop.v2` binary transport, frame drop metrics, ordered async encode pool. **JPEG rects use libjpeg-turbo** (SIMD, direct BGRA input — no RGBA pre-pass; 4:4:4 chroma on High/Medium tiers so colored text stays crisp; pure-Go `image/jpeg` 4:2:0 fallback if libturbojpeg is unavailable — watch the `jpeg_turbo` field of the encoder-pool init log). Browser zstd_bgra decode prefers native `DecompressionStream("zstd")` (wasm fallback); BGRA rects on the Canvas-2D surface convert via `VideoFrame(BGRX)` (native, zero JS swizzle) with a word-wise JS swizzle fallback. |
| Keyboard/mouse input | Enabled | FreeRDP owner-thread dispatch. **Scancode-based**: the browser maps `event.code` (physical key) → RDP set-1 scancode (+extended) for ALL keys, so modifier shortcuts compose on the server — Ctrl+C/V/X/Z, Alt+Tab, Win+L, Ctrl+Shift+Esc, Alt+F4 all work (Unicode injection, used before for printable keys, can't form combos). The server's keyboard layout (`FreeRDP_KeyboardLayout` ← keyboardLayout setting) turns the scancode into the right character, so set it to match the physical keyboard. Held scancodes are released on window blur / tab-hide so an OS-reserved combo (Alt+Tab, Win+L) stealing focus can't leave a modifier stuck down. Toolbar/palette "send combo" buttons emit the same scancodes for OS-reserved shortcuts the browser can't intercept. Keys without a scancode mapping fall back to keysym → scancode/Unicode. |
| IME / CJK text input | Enabled | The browser keeps a hidden editable sink (`data-desktop-passthrough` textarea, positioned at the last click so the candidate window pops near the cursor); the OS input method composes there. Raw keystrokes are suppressed while composing (`keysymForEvent` skips `isComposing` / `keyCode 229` / `key "Process"`), and the committed phrase is sent on `compositionend` as `ClientMessage.text`. The worker replays it as per-character Unicode keyboard events (UTF-16 code units, surrogate pairs for astral chars) so it lands regardless of the remote keyboard layout — client-side IME, like mstsc. The commit key's trailing keyup is swallowed so no stray key leaks after the text. Recorded to the input-audit tape. |
| Text clipboard | Enabled | `CF_UNICODETEXT` round trip (in/out) |
| Graphics pipeline (RDPGFX) | Enabled | Surface lifecycle + frame ack; decoded into the GDI primary buffer and re-emitted as BGRA/JPEG/zlib, or forwarded as AVC420 H.264. Gated off automatically when the browser lacks `WebCodecs.VideoDecoder` (`StartSessionRequest.ClientCaps`). |
| H.264 / AVC420 (GPU decode) | Enabled | Browser decodes via `WebCodecs.VideoDecoder` (`avc1.42E01E`, Annex-B in-band SPS/PPS). Keyframe flag drives `key`/`delta` chunks; decoder errors request an RDP Refresh Rect. AVC444 stays off (browser decodes a single stream only). Used by the **bitmap path**; the WebRTC path below supersedes it when active. |
| WebRTC video (VP9/VP8, GPU decode) | Enabled (freerdp) | Default video transport for WebRTC-capable browsers (`desktop.webrtc.enabled`, browser reports `ClientCaps.webrtc`). The worker disables the RDPGFX pipeline, lets FreeRDP's GDI composite every codec into `primary_buffer`, encodes that with libvpx (realtime CBR), and emits access units the gateway feeds to a Pion `TrackLocalStaticSample`. The browser renders a `<video>` element (hardware decode) — no JS decode, no dirty-bitmap blits. **Codec: VP9 screen-content** (`VP9E_CONTENT_SCREEN` — palette/intra-block-copy coding purpose-built for desktop text/UI, much sharper than VP8 at equal bitrate) is the default (`desktop.webrtc.codec`), used when the browser advertises VP9 decode (`ClientCaps.webrtcVP9`); VP8 is the fallback. Signaling (browser offer → gateway answer, trickle ICE) rides the existing desktop WS; ICE is host-only by default with optional STUN/TURN/public-IP (`desktop.webrtc.*`). RTCP PLI/FIR → worker keyframe request. **Low latency**: the browser shrinks the receiver jitter buffer (`jitterBufferTarget`/`playoutDelayHint`→0). **User choice**: the desktop settings sheet lets the user force WebRTC, force JS bitmap, or auto (`StartSessionRequest.video_transport`), and pick a quality tier (`video_quality` → bitrate ×0.5/×1/×2). Falls back to the bitmap path automatically (worker `video_mode: "bitmap"`) when the browser lacks WebRTC or negotiation fails; "forced WebRTC" retries WebRTC on reconnect, "auto" sticks to bitmap after a failure. Never recorded over WebRTC (recording uses the bitmap tape). **Encoder tuning (libvpx/libaom realtime)**: threads + tile-columns scale with host cores and surface width; VP9/AV1 run cyclic-refresh AQ (`AQ_MODE 3`); keyframe size capped at 450% of the average frame budget so PLI-forced IDRs can't stall a starved link; VP8 fallback enables `SCREEN_CONTENT_MODE`; AV1 additionally disables the TPL model and lowers entropy-cost update frequency (RTC-example parity). **Transport**: encoded access units cross the worker→gateway stdout pipe as `BinaryFrameVideo` frames (raw bytes; the old JSON+base64 hop cost ~1.3 MB/s of churn at 8 Mbps). |
| Proxy-chain forwarding | Enabled | The worker's TCP transport is routed through the node's `proxy_chain` (SSH bastion / SOCKS5 hops) via a per-session gateway-local SOCKS5 listener (`internal/socks5`), the same path guacamole/tcpfwd use. Nodes with no `proxy_chain` dial directly. |
| RD Gateway (MS-TSGU) | Enabled (freerdp) | Reaches targets published only through a Microsoft Remote Desktop Gateway (HTTPS tunnel). Configured per node (`rdp.gateway_host/port/domain/transport`): the worker sets `FreeRDP_GatewayEnabled` + `GatewayUsageMethod=TSC_PROXY_MODE_DIRECT` + `GatewayHostname/Port` and the transport (`auto` = HTTP/WebSocket then RPC fallback; `http` = WebSocket only; `rpc` = RPC-over-HTTP only). Auth: `gateway_use_same_credentials` (default) reuses the target login via `FreeRDP_GatewayUseSameCredentials`; otherwise a dedicated **sealed** credential (`gateway_credential_id`, resolved by the manager) supplies `GatewayUsername/Password/Domain` — the gateway password never lands in `proto_options`. Independent of and composable with the proxy-chain (FreeRDP routes the gateway's own TCP through the SOCKS proxy when both are set). Configured in the node's RDP options (网络 tab). ironrdp backend: not wired (use freerdp). |
| RemoteFX (RFX) | Off by default | Negotiable per node (`enable_remote_fx`) but **the browser has no RFX decoder** — an RFX frame is skipped (logged), not rendered. Leave disabled until a browser-side RFX decoder is wired. |
| Dynamic resize (RDPEDISP) | Disabled | DISP channel callback, browser resize ack, reconnect fallback. The browser viewport is recorded; the new size takes effect on reconnect. |
| Audio playback (RDPSND) | Enabled (freerdp) | Custom in-worker rdpsnd device ("jsaudio" subsystem, installed via a chained FreeRDP addin provider) advertises PCM-only and forwards the negotiated PCM to the browser, which plays it via Web Audio (`audio-player.ts`, 60ms jitter buffer). Offered as both static + dynamic rdpsnd (modern Windows uses the dynamic `AUDIO_PLAYBACK_DVC`). Enabled whenever drive redirection is on — rdpsnd presence is also what makes Windows initialise its RDPDR subsystem. Gated browser-side by the `audioPlayback` setting; never written to recordings. |
| Drive redirection (RDPDR) | Enabled (freerdp) | Per-user persistent folder (`<sessions_dir>/desktop-drives/user-<id>`) redirected as a mounted drive via the `rdpdr`+`drive` static addins. Browser file panel (`/desktop/drive/*`) lists/uploads/downloads/deletes against the same folder, so uploads appear in the remote "This PC" instantly. Upload/download directions, per-file and per-user size caps are configurable (`desktop.drive`); transfers write `file.upload`/`file.download` audit events. Not available on the ironrdp backend (no per-session worker folder). |
| Printers/smartcards | Disabled | explicit security review and operator policy |
| ironrdp backend proxy chain | Direct-dial only | The Devolutions Gateway subprocess opens the backend TCP connection itself; it is **not** routed through the JumpServer proxy chain. Use the freerdp backend for bastion-only nodes. |

Disabled capabilities must remain forced off in `cmd/freerdp-worker/rdp/client.go` until the worker, gateway, browser protocol, UI, audit, and operator policy are implemented end to end.
